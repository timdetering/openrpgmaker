  • Version 3 to 4
    • Number of phases has been reduced to 24
      • This is because the map data has gotten very complex and uses 5 bits for each phase (1 for passability and 4 for the 4 directions).
      • The reason 32 phases was chosen initially was because each phase used only 1 bit on the map and could thus be saved using a single 32bit unsigned long (8 hexadecimal characters, with the tile data using 6, making a total of 14 characters per tile)
      • The reason for 24 phases now is because each phase takes 5 bits (for a total of 15 bytes) and the tile data takes 3 bytes (tilesetID, tileID, and tileFlags) for a total of 18 bytes. 18 bytes can be evenly encoded in base64 using 24 characters (hexadecimal for the original 32 phases would take 46 characters, or 36 characters for 24 phases). As you can see 24 characters is the most optimal for storage space while still allowing 16 phases to be defined by the user (8 are used internally by the system for built-in mechanics such as jumping).
    • map*.xml
      • for each layer
        • iterate over every tile and remove the last eight phase passabilities
        • then for each phase calculate the directional passability based on neighboring tiles
        • then convert the layer data to base64 encoding instead of hexadecimal
          NOTE: Map data has gotten too complex to continue to use the more human readable hexadecimal format, because it would take too much space.
    • tileset*.xml
      • for each tile
        • remove the last eight phase passabilities from the "phasePassability" attribute
        • add the "phaseDirections" attribute with each phase that is on having all four directions set.
    • vehicles*.xml
      • If the vehicle's "passingPhase" attribute is greater than 23 set it to 23 (only 24 phases available now)
      • If the vehicle's "landingPhase" attribute is greater than 23 set it to 23 (only 24 phases available now)
